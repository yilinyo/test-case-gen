# 函数测试用例自动生成工具需求文档

## 1. 项目概述

本项目实现一个函数级测试用例自动生成工具。用户提供 Python 函数、参数范围、测试规则或自然语言需求后，工具能够自动生成结构化测试用例，并可选执行被测函数，输出实际结果、期望结果和通过状态。

工具面向软件测试课程实验、小型函数回归测试和自动化测试原型验证。

## 2. 用户角色

- 测试人员：根据函数需求快速生成测试用例和测试报告。
- 开发人员：对已有函数进行分支覆盖和回归验证。
- 学生或教师：用于软件测试课程中的实验演示和报告生成。

## 3. 功能需求

### 3.1 规则模式

工具应支持读取用户编写的测试规格 JSON 文件。

测试规格应包含：

- 被测函数名称；
- 被测函数位置；
- 参数名称、类型、范围和候选值；
- 测试规则列表；
- 每条规则的场景编号、场景名称、触发条件和期望结果。

工具应根据规则确定性生成测试用例。

### 3.2 代码分析模式

工具应支持根据被测函数源码自动生成基础测试规格。

初版代码分析范围：

- 支持 Python 函数；
- 支持顺序 `if / return` 结构；
- 支持默认 `return` 分支；
- 能够根据函数分支生成测试场景；
- 能够结合参数范围枚举输入数据。

该模式生成的期望结果来自当前代码返回值，主要用于白盒测试和回归测试。

### 3.3 大模型模式

工具应支持调用 OpenAI 兼容大模型接口。

大模型模式应完成：

- 读取函数源码；
- 读取自然语言需求说明；
- 构造测试规格生成 Prompt；
- 调用大模型生成测试规格 JSON；
- 校验大模型返回内容；
- 根据校验后的测试规格生成测试用例。

大模型不得直接生成最终测试结果，最终用例生成和函数执行必须由本地确定性程序完成。

### 3.4 混合模式

工具应支持生成大模型 Prompt 文件，供用户复制到模型中生成测试规格草稿。

用户确认或修改草稿后，可使用规则模式继续生成测试用例。

混合模式用于兼顾大模型生成效率和人工确认准确性。

### 3.5 测试规格校验

工具应校验测试规格文件。

校验内容包括：

- JSON 格式合法；
- 必要字段存在；
- 参数名合法；
- 规则编号不重复；
- `when` 表达式语法合法；
- `when` 表达式只能引用已声明参数；
- 禁止函数调用、属性访问、下标访问、导入、文件操作和系统命令。

### 3.6 测试用例生成

工具应根据参数配置生成候选输入。

对于整数参数，应支持：

- 最小值；
- 最大值；
- 边界值；
- 典型值；
- 用户指定候选值；
- 小范围整数完整枚举。

工具应对每条规则筛选满足条件的输入，并生成测试用例。

每条规则最多生成的用例数量应可配置。

### 3.7 函数执行

当测试规格或命令行提供被测函数位置时，工具应动态导入并执行函数。

执行后应记录：

- 输入参数；
- 期望结果；
- 实际结果；
- 是否通过；
- 异常信息。

### 3.8 报告导出

工具应支持以下输出格式：

- CSV；
- JSON；
- Markdown。

报告内容应包含：

- 用例编号；
- 函数名称；
- 场景编号；
- 场景名称；
- 触发条件；
- 输入参数；
- 期望结果；
- 实际结果；
- 通过状态；
- 错误信息。

## 4. 非功能需求

### 4.1 可复现性

同一份测试规格在相同参数配置下应生成稳定一致的测试用例。

### 4.2 安全性

规则表达式必须通过 AST 白名单校验。

工具不应直接执行未经限制的规则字符串。

### 4.3 易用性

工具应提供命令行入口，并支持清晰的参数提示。

示例命令应覆盖规则模式、代码分析模式、大模型模式和混合模式。

### 4.4 可扩展性

工具应采用模块化结构，便于后续扩展：

- 更多数据类型；
- 更复杂的 AST 分析；
- pytest 文件导出；
- Web 界面；
- 更多大模型供应商；
- 沙箱执行被测函数。

## 5. 输入要求

### 5.1 被测函数

被测函数应为 Python 函数。

函数位置格式：

```text
module:function
```

或：

```text
path/to/file.py:function
```

### 5.2 测试规格文件

测试规格文件应为 JSON 格式。

核心结构：

```json
{
  "function": "函数名",
  "target": "模块:函数",
  "max_cases_per_rule": 3,
  "params": {
    "参数名": {
      "type": "int",
      "min": -10,
      "max": 10,
      "values": [0, 1, 2]
    }
  },
  "rules": [
    {
      "id": "场景编号",
      "name": "场景名称",
      "when": "参数条件表达式",
      "expect": "期望结果"
    }
  ]
}
```

### 5.3 参数配置文件

参数配置文件用于代码分析模式。

示例：

```json
{
  "params": {
    "d1": {
      "type": "int",
      "min": -1,
      "max": 10
    }
  }
}
```

### 5.4 需求说明文件

需求说明文件用于大模型模式和混合模式。

文件内容可以是自然语言，应描述函数输入、业务规则和期望返回值。

## 6. 输出要求

### 6.1 CSV 输出

CSV 输出适合导入 Excel 或作为课程报告附件。

### 6.2 JSON 输出

JSON 输出适合后续程序读取或接入自动化测试平台。

### 6.3 Markdown 输出

Markdown 输出适合直接放入实验报告或项目文档。

## 7. 命令行需求

工具入口：

```bash
python3 testcase_generator.py
```

规则模式：

```bash
python3 testcase_generator.py \
  --mode rule \
  --spec examples/triangle_spec.json \
  --out outputs/triangle_cases.csv
```

代码分析模式：

```bash
python3 testcase_generator.py \
  --mode auto \
  --call examples.triangle_impl:check_triangle \
  --params examples/triangle_params.json \
  --out outputs/triangle_auto_cases.md \
  --out-spec outputs/triangle_auto_spec.json
```

混合模式：

```bash
python3 testcase_generator.py \
  --mode hybrid \
  --call examples.triangle_impl:check_triangle \
  --requirement examples/triangle_requirement.md \
  --out-prompt outputs/triangle_llm_prompt.md
```

大模型模式：

```bash
python3 testcase_generator.py \
  --mode llm \
  --call examples.triangle_impl:check_triangle \
  --requirement examples/triangle_requirement.md \
  --out-spec outputs/triangle_llm_spec.json \
  --out outputs/triangle_llm_cases.json
```

## 8. 验收标准

项目应满足以下验收条件：

- 能够运行规则模式并生成 CSV 测试报告；
- 能够运行代码分析模式并生成测试规格和 Markdown 报告；
- 能够运行混合模式并生成大模型 Prompt 文件；
- 当配置大模型 API Key 时，能够运行大模型模式；
- 生成的三角形函数测试用例覆盖非法边长、三角形不等式失败、等边、等腰和普通三角形；
- 测试报告中包含 expected、actual 和 pass 字段；
- 不合法规则表达式会被拒绝；
- 项目 README 中提供可执行示例命令。

## 9. 当前限制

- 代码分析模式当前主要支持简单顺序 `if / return` 函数；
- 被测函数执行默认信任本地代码，尚未提供沙箱隔离；
- 规则表达式不支持函数调用、属性访问和下标访问；
- 大模型生成质量依赖需求说明质量和所使用模型；
- 暂未生成 pytest 文件。

## 10. 后续扩展

- 支持嵌套分支和 `elif / else` 更精细路径分析；
- 支持列表、字典、枚举等复杂参数类型；
- 支持 pytest 测试代码导出；
- 支持 Web 页面交互式生成测试用例；
- 支持被测函数超时控制和子进程隔离；
- 支持更多大模型服务商配置。
