# 基于大模型的函数测试用例自动生成工具设计方案

## 1. 项目名称

**基于大模型与功能场景规则的函数测试用例自动生成工具**

也可以命名为：

- FuncCaseGen
- SmartTestCaseGen
- LLM-TestCase-Generator
- Function Scenario Test Generator

---

## 2. 项目背景

在软件测试课程或实际项目中，开发者经常需要针对函数设计测试用例。例如三角形判断函数：

```python
def check_triangle(d1: int, d2: int, d3: int) -> str:
    ...
```

传统方式通常由人工根据需求编写测试用例，这种方式存在以下问题：

1. **人工设计成本高**  
   需要逐条分析函数功能、边界条件和异常情况。

2. **容易遗漏边界场景**  
   例如三角形问题中，很多人只会测试 `0`、负数、等边、等腰，却容易遗漏 `(1, 2, 3)` 这种不满足三角形不等式的情况。

3. **测试用例复用性差**  
   不同函数需要重新设计测试数据，缺少统一的测试规格描述方式。

4. **无法快速生成结构化报告**  
   手工编写的测试用例往往不方便导出为 CSV、JSON、Markdown 或接入自动化测试框架。

因此，本项目设计一个支持**规则模式、代码分析模式、大模型模式、混合模式**的函数测试用例自动生成工具，帮助用户根据函数源码和需求说明自动生成测试规格、测试用例和测试结果报告。

---

## 3. 项目目标

本工具的核心目标是：  
**根据函数源码、参数范围、自然语言需求或规则配置，自动生成覆盖主要功能场景的测试用例，并可选执行函数验证结果。**

具体目标包括：

1. **支持规则驱动生成**
   - 用户手动编写 `rules`；
   - 工具根据规则生成测试用例；
   - 适合黑盒测试和功能测试。

2. **支持代码分析生成**
   - 工具自动分析函数源码中的 `if / elif / else / return`；
   - 生成分支覆盖测试用例；
   - 适合白盒测试和回归测试。

3. **支持大模型辅助生成**
   - 大模型读取函数源码和需求说明；
   - 自动生成测试规格 JSON；
   - 工具再根据测试规格生成测试用例；
   - 降低人工编写规则的成本。

4. **支持混合模式**
   - 大模型先生成规则草稿；
   - 用户可以修改或确认；
   - 工具再生成最终测试用例；
   - 兼顾智能化和准确性。

5. **支持测试执行与报告导出**
   - 自动导入被测函数；
   - 执行测试用例；
   - 比较 expected 和 actual；
   - 输出 CSV、JSON、Markdown 测试报告。

---

## 4. 总体设计思想

本工具不直接让大模型生成最终测试结果，而是采用更稳定的两阶段设计：

```text
第一阶段：生成测试规格
    ↓
第二阶段：根据测试规格确定性生成测试用例
```

其中，大模型只负责辅助生成 `test_spec.json`，真正的测试用例生成、函数执行和结果比较由确定性的 Python 程序完成。

这样设计的好处是：

1. **降低大模型幻觉影响**
   - 大模型只生成规则草稿；
   - 工具会校验 JSON 格式、参数名和规则表达式。

2. **便于复现**
   - 同一个 `test_spec.json` 可以稳定生成相同测试用例。

3. **便于调试**
   - 如果测试用例不合理，可以直接修改 `rules`。

4. **便于扩展**
   - 后续可以接入 pytest、Web 页面、CI/CD 系统。

## 6. 工作模式设计

### 6.1 规则模式：`rule`

规则模式要求用户手动编写测试规格文件，例如：

```bash
python3 testcase_generator.py \
  --mode rule \
  --spec triangle_spec.json \
  --out triangle_cases.csv
```

适用场景：

- 用户已经明确函数需求；
- 需要严格按照规格进行黑盒测试；
- 测试结果要和真实需求对比。

优点：

- 准确性最高；
- expected 来自人工规格；
- 能发现代码实现是否符合需求。

缺点：

- 需要人工编写规则；
- 对初学者有一定门槛。

---

### 6.2 代码分析模式：`auto`

代码分析模式不需要用户手写 rules，而是自动分析函数源码。

示例命令：

```bash
python3 testcase_generator.py \
  --mode auto \
  --call triangle_impl:check_triangle \
  --params triangle_params.json \
  --out triangle_cases.csv
```

工具会做以下事情：

1. 使用 `inspect.getsource()` 获取函数源码；
2. 使用 `ast.parse()` 解析语法树；
3. 提取 `if / elif / else / return` 结构；
4. 生成分支路径；
5. 枚举参数候选值；
6. 找到能够覆盖每个分支的输入；
7. 执行函数，得到实际返回值。

适用场景：

- 白盒测试；
- 分支覆盖测试；
- 回归测试；
- 快速了解函数行为。

优点：

- 不需要写 rules；
- 能自动覆盖代码分支；
- 适合测试已有代码。

缺点：

- expected 来自当前代码；
- 如果代码本身写错，工具也可能把错误结果当成 expected；
- 不能完全代表真实业务需求。

---

### 6.3 大模型模式：`llm`

大模型模式是本项目的重点扩展模式。

示例命令：

```bash
python3 testcase_generator.py \
  --mode llm \
  --call triangle_impl:check_triangle \
  --requirement triangle_requirement.md \
  --out-spec triangle_spec_llm.json
```

或者一步生成测试用例：

```bash
python3 testcase_generator.py \
  --mode llm \
  --call triangle_impl:check_triangle \
  --requirement triangle_requirement.md \
  --out triangle_cases_with_result.csv
```

大模型模式流程：

```text
读取函数源码
    ↓
读取自然语言需求说明
    ↓
构造 Prompt
    ↓
调用大模型
    ↓
生成 test_spec.json
    ↓
校验 test_spec.json
    ↓
生成测试用例
    ↓
可选执行函数
    ↓
输出测试报告
```

适用场景：

- 用户不想手写规则；
- 需求说明是自然语言；
- 希望自动补充边界场景；
- 课程项目需要体现智能化设计。

优点：

- 降低规则编写成本；
- 能根据需求说明生成测试场景；
- 能自动补充正常、异常、边界情况；
- 项目亮点更明显。

缺点：

- 大模型可能误解需求；
- 大模型可能生成错误规则；
- 需要增加规则校验和用户确认流程。

---

### 6.4 混合模式：`hybrid`

混合模式是最推荐的最终形态。

流程如下：

```text
函数源码 + 需求说明
    ↓
大模型生成 rules 草稿
    ↓
工具校验 rules
    ↓
用户人工确认或修改
    ↓
生成最终测试用例
    ↓
执行函数并输出报告
```

示例命令：

```bash
python3 testcase_generator.py \
  --mode hybrid \
  --call triangle_impl:check_triangle \
  --requirement triangle_requirement.md \
  --out-spec draft_spec.json
```

用户修改 `draft_spec.json` 后，再执行：

```bash
python3 testcase_generator.py \
  --mode rule \
  --spec draft_spec.json \
  --out final_cases.csv \
  --call triangle_impl:check_triangle
```

混合模式兼顾了：

- 大模型的自动化能力；
- 人工规格确认的准确性；
- 程序生成测试用例的稳定性。

## 7. 核心数据结构设计

### 7.1 测试规格文件 `test_spec.json`

测试规格文件是整个系统的核心中间表示。

示例：

```json
{
  "function": "Check_Triangle",
  "target": "triangle_impl:check_triangle",
  "max_cases_per_rule": 3,
  "params": {
    "d1": {
      "type": "int",
      "min": -1,
      "max": 10,
      "values": [1, 2, 3, 4, 5]
    },
    "d2": {
      "type": "int",
      "min": -1,
      "max": 10,
      "values": [1, 2, 3, 4, 5]
    },
    "d3": {
      "type": "int",
      "min": -1,
      "max": 10,
      "values": [1, 2, 3, 4, 5]
    }
  },
  "rules": [
    {
      "id": "non_positive",
      "name": "边长小于等于 0，非三角形",
      "when": "d1 <= 0 or d2 <= 0 or d3 <= 0",
      "expect": "non-triangle"
    },
    {
      "id": "triangle_inequality_invalid",
      "name": "不满足三角形不等式，非三角形",
      "when": "d1 > 0 and d2 > 0 and d3 > 0 and (d1 + d2 <= d3 or d1 + d3 <= d2 or d2 + d3 <= d1)",
      "expect": "non-triangle"
    },
    {
      "id": "equilateral",
      "name": "三边相等，等边三角形",
      "when": "d1 > 0 and d2 > 0 and d3 > 0 and d1 + d2 > d3 and d1 + d3 > d2 and d2 + d3 > d1 and d1 == d2 and d2 == d3",
      "expect": "equilateral triangle"
    },
    {
      "id": "isosceles",
      "name": "有效三角形且两边相等，等腰三角形",
      "when": "d1 > 0 and d2 > 0 and d3 > 0 and d1 + d2 > d3 and d1 + d3 > d2 and d2 + d3 > d1 and (d1 == d2 or d1 == d3 or d2 == d3) and not (d1 == d2 and d2 == d3)",
      "expect": "isosceles triangle"
    },
    {
      "id": "scalene",
      "name": "有效三角形且三边互不相等，普通三角形",
      "when": "d1 > 0 and d2 > 0 and d3 > 0 and d1 + d2 > d3 and d1 + d3 > d2 and d2 + d3 > d1 and d1 != d2 and d1 != d3 and d2 != d3",
      "expect": "other triangle"
    }
  ]
}
```

### 7.2 参数配置文件 `params.json`

在代码分析模式中，用户可能不想写完整 rules，但仍然需要告诉工具参数范围。

示例：

```json
{
  "params": {
    "d1": {
      "type": "int",
      "min": -1,
      "max": 10
    },
    "d2": {
      "type": "int",
      "min": -1,
      "max": 10
    },
    "d3": {
      "type": "int",
      "min": -1,
      "max": 10
    }
  }
}
```

---

### 7.3 需求说明文件 `requirement.md`

大模型模式建议提供自然语言需求说明。

示例：

```markdown
# 三角形判断函数需求

函数 `check_triangle(d1, d2, d3)` 用于判断三条边能够构成哪种三角形。

规则如下：

1. 如果任意边长小于等于 0，则返回 `non-triangle`。
2. 如果三条边不满足三角形两边之和大于第三边，则返回 `non-triangle`。
3. 如果三边相等，则返回 `equilateral triangle`。
4. 如果只有两边相等，则返回 `isosceles triangle`。
5. 如果三边互不相等且构成三角形，则返回 `other triangle`。
```

## 8. 大模型模块设计

### 8.1 大模型模块职责

大模型模块不直接执行测试，也不直接判断测试是否通过。它只负责：

1. 阅读函数源码；
2. 阅读需求说明；
3. 识别函数参数；
4. 识别功能场景；
5. 生成测试规格 JSON；
6. 补充边界场景和异常场景。

也就是说，大模型输出的是：

```text
test_spec.json
```

而不是最终测试结果。

---

### 8.2 Prompt 模板设计

大模型 Prompt 可以设计如下：

```text
你是一个软件测试用例设计助手。

请根据以下函数源码和需求说明，生成函数测试规格 JSON。

要求：
1. 输出必须是合法 JSON；
2. 不要输出 Markdown；
3. 不要添加解释文字；
4. 每个测试场景必须包含 id、name、when、expect；
5. when 字段必须是 Python 布尔表达式；
6. rules 必须覆盖正常情况、异常情况、边界情况；
7. 参数名必须与函数参数一致；
8. 不要使用函数调用、属性访问、下标访问；
9. 不要生成工具不支持的表达式；
10. 如果需求说明和函数源码冲突，以需求说明为准。

函数源码：
{function_source}

需求说明：
{requirement}

输出 JSON 格式：
{
  "function": "...",
  "target": "...",
  "max_cases_per_rule": 3,
  "params": {
    "参数名": {
      "type": "int",
      "min": -10,
      "max": 10
    }
  },
  "rules": [
    {
      "id": "...",
      "name": "...",
      "when": "...",
      "expect": "..."
    }
  ]
}
```

---

### 8.3 大模型输出校验

大模型输出不能直接信任，必须经过校验。

校验流程如下：

```text
LLM 原始输出
    ↓
提取 JSON
    ↓
JSON 格式校验
    ↓
必要字段校验
    ↓
参数名一致性校验
    ↓
when 表达式 AST 安全校验
    ↓
规则可满足性校验
    ↓
测试用例生成
```

校验内容包括：

| 校验项 | 说明 |
|---|---|
| JSON 格式 | 是否能被 `json.loads()` 解析 |
| 必要字段 | 是否包含 `function`、`params`、`rules` |
| 参数名 | `when` 中的变量是否都在 `params` 中 |
| 表达式安全 | 禁止函数调用、属性访问、文件操作等 |
| 规则可满足性 | 每条 rule 是否能找到输入样例 |
| expected 类型 | 是否是字符串、数字或布尔值 |
| 场景覆盖 | 是否包含正常、异常、边界情况 |

## 9. 代码分析模块设计

代码分析模式主要基于 Python AST 实现。

### 9.1 使用的核心库

```python
import inspect
import ast
```

### 9.2 分析流程

```text
导入目标函数
    ↓
inspect.getsource() 获取源码
    ↓
ast.parse() 转换为语法树
    ↓
遍历 ast.If 节点
    ↓
提取 test 条件
    ↓
提取 return 返回值
    ↓
构造路径条件
    ↓
枚举参数输入
    ↓
找到覆盖分支的测试用例
```

### 9.3 能支持的代码结构

初版建议支持简单函数：

```python
def func(...):
    if 条件1:
        return 结果1

    if 条件2:
        return 结果2

    if 条件3:
        return 结果3

    return 默认结果
```

后续再扩展支持：

- `elif`;
- `else`;
- 嵌套 `if`;
- 异常抛出；
- 多返回类型；
- 循环结构。

## 10. 测试用例生成模块设计

### 10.1 候选值生成

对于 int 类型参数，候选值包括：

```text
min
min + 1
-1
0
1
中间值
max - 1
max
用户指定 values
```

如果范围较小，可以完整枚举。

例如 `[-1, 10]`，可以生成：

```text
-1, 0, 1, 2, 3, ..., 10
```

### 10.2 规则匹配

工具会遍历候选输入组合：

```text
for each rule:
    for each input_combination:
        if rule.when == True:
            create test case
```

### 10.3 去重策略

同一组输入如果已经被某条规则使用，则不重复加入其他场景，避免测试用例重复。

### 10.4 每个场景数量控制

通过字段控制：

```json
"max_cases_per_rule": 3
```

表示每条规则最多生成 3 条测试用例。

## 13. 三角形函数示例

### 13.1 被测函数

```python
def check_triangle(d1: int, d2: int, d3: int) -> str:
    if d1 <= 0 or d2 <= 0 or d3 <= 0:
        return "non-triangle"

    if d1 + d2 <= d3 or d1 + d3 <= d2 or d2 + d3 <= d1:
        return "non-triangle"

    if d1 == d2 and d2 == d3:
        return "equilateral triangle"

    if d1 == d2 or d1 == d3 or d2 == d3:
        return "isosceles triangle"

    return "other triangle"
```

### 13.2 需求说明

```text
输入三条边 d1、d2、d3。
如果任意边小于等于 0，返回 non-triangle。
如果不满足三角形两边之和大于第三边，返回 non-triangle。
如果三边相等，返回 equilateral triangle。
如果只有两边相等，返回 isosceles triangle。
否则返回 other triangle。
```

### 13.3 大模型应生成的场景

| 场景 | 示例输入 | 期望输出 |
|---|---|---|
| 边长小于等于 0 | `(0, 3, 4)` | `non-triangle` |
| 不满足三角形不等式 | `(1, 2, 3)` | `non-triangle` |
| 等边三角形 | `(3, 3, 3)` | `equilateral triangle` |
| 等腰三角形 | `(2, 2, 3)` | `isosceles triangle` |
| 普通三角形 | `(3, 4, 5)` | `other triangle` |

## 18. 安全性设计

### 18.1 规则表达式安全

由于 `when` 字段需要执行布尔表达式，所以必须限制表达式能力。

允许：

- 变量名；
- 常量；
- `and`;
- `or`;
- `not`;
- 比较运算；
- 基础算术运算。

禁止：

- 函数调用；
- 属性访问；
- 文件操作；
- 系统命令；
- import；
- exec；
- eval；
- 下标访问。

### 18.2 被测函数执行风险

执行用户函数本身也可能有风险。初版课程项目可以默认本地可信代码，后续可以增加：

- 子进程隔离；
- 超时控制；
- 禁止网络；
- Docker 沙箱；
- 临时目录执行。

## 21. 风险与解决方案

| 风险 | 说明 | 解决方案 |
|---|---|---|
| 大模型生成错误规则 | LLM 可能误解需求 | 加规则校验和人工确认 |
| expected 不准确 | 仅根据代码生成 expected 可能继承代码错误 | 优先使用需求说明 |
| 规则表达式不安全 | `when` 需要被执行 | 使用 AST 白名单校验 |
| 参数范围过大 | 笛卡尔积组合爆炸 | 限制 max_total_search，支持边界值策略 |
| 函数执行不安全 | 用户函数可能有副作用 | 后续加入沙箱和超时 |
| 分支分析能力有限 | AST 初版只能处理简单 if | 逐步支持嵌套、elif、异常等结构 |

## 23. 总结

本方案设计了一个基于大模型与功能场景规则的函数测试用例自动生成工具。工具支持规则模式、代码分析模式、大模型模式和混合模式。

其中，大模型模式不是直接生成最终测试结果，而是生成结构化测试规格 `test_spec.json`。随后，工具通过确定性的规则校验、候选值生成、规则匹配和函数执行流程生成最终测试用例和测试报告。

这种设计既能降低人工编写测试用例的成本，又能保证测试生成过程具备一定的可解释性和可复现性。以三角形判断函数为例，该工具能够自动覆盖边长非法、不满足三角形不等式、等边三角形、等腰三角形和普通三角形等典型功能场景，适合作为软件测试课程项目、实验报告或函数级自动化测试平台原型。
